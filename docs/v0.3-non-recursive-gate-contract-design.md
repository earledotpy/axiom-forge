# v0.3 Design — Non-Recursive Gate-Contract Test Architecture

## Purpose and scope

This is a design-only proposal for a future gate-contract test architecture.
It does not change the configured promotion-time verification command, promotion
policy, adapter status, or any current script. It is grounded in the tracked
implementation at `0cdee9d` and the deferred candidate in `docs/v0.2-progress.md`.

The design goal is to provide future regression coverage for the runner and
promotion contracts without allowing promotion-time verification to invoke
promotion again. The proposed architecture is not an orchestration layer,
trust system, or automatic-promotion mechanism.

## Current enforced invariants

### Runner invariants

| Invariant | Current enforcement or evidence |
| --- | --- |
| The target repository is clean before a run begins. | `scripts/run_agent_task.sh` rejects `target_repo_dirty`; `tests/runner/run_all.sh` requires a clean preflight. |
| The adapter name is safe and the adapter script exists. | `scripts/run_agent_task.sh` checks `safe_run_id` and records `agent_adapter_not_found`; runner case R1 covers the missing adapter. |
| The agent receives a disposable detached worktree at the captured base SHA. | `scripts/run_agent_task.sh` creates and later removes `agent-<run-id>` with `git worktree add --detach`. |
| The target checkout does not change outside the supplied worktree during adapter execution. | `scripts/run_agent_task.sh` compares before/after Git status including ignored paths and records `adapter_modified_outside_worktree`; runner cases R2a–R2e cover tracked and ignored parent-checkout writes, non-completion, and cleanup. This is detection, not OS containment. |
| An adapter must return successfully before a run can continue. | `scripts/run_agent_task.sh` records `agent_execution_failed` for a nonzero adapter result; runner case R5 covers this path. |
| An adapter must not change the worktree `HEAD`, leave detached `HEAD`, or create/delete branches. | `scripts/run_agent_task.sh` records `adapter_changed_head`, `adapter_left_detached_head`, or `adapter_created_or_deleted_branch`; runner cases R2–R3 cover these paths. |
| A captured run must have a non-empty patch and a computed patch digest before completion. | `scripts/run_agent_task.sh` records `agent_produced_empty_patch` or `patch_sha256_compute_failed`; runner cases R4 and R6 cover empty and valid output. |
| Failed runs remain records and successful runs validate before later use. | `scripts/write_run_record.py`, `scripts/validate_run_dir.sh`, and runner cases R1–R7 provide current evidence. |

### Verifier invariants

| Invariant | Current enforcement or evidence |
| --- | --- |
| A promotable run directory must exist, contain `record.json` and a non-empty patch, use a safe ID, and bind `record.json.run_id` to its directory name. | `scripts/validate_run_dir.sh`; promotion cases T1–T4c include missing inputs and `run_id_directory_mismatch`. |
| A run must be `COMPLETED`, name an existing base commit, and match its recorded patch SHA-256 when present. | `scripts/validate_run_dir.sh`. |
| Verification applies the recorded patch from its recorded base in a fresh detached worktree. | `scripts/verify_patch.sh` creates a temporary detached worktree and removes it by trap cleanup. |
| The patch must pass `git apply --check --whitespace=error` and apply successfully. | `scripts/verify_patch.sh`; promotion cases T6–T7 exercise non-applying and whitespace-invalid patches. |
| Configured required checks execute with the configured timeout and produce a structured verification result. | `scripts/verify_target.py`; `gate.toml` currently configures only `unit`. |

### Promotion invariants

| Invariant | Current enforcement or evidence |
| --- | --- |
| Promotion begins only from a clean target repository. | `scripts/promote.sh` records `target_repo_dirty`; promotion matrix preflight requires cleanliness. |
| The recorded base must equal the current configured default-base SHA. | `scripts/promote.sh` records `stale_base_sha`; cases T5a–T5d cover valid historical structure, rejection, recorded reason, and no branch. |
| The proposed gate branch name is valid and must not already exist. | `scripts/promote.sh`; case T9 covers the existing branch path. |
| The patch is re-verified before approval and after the promotion commit. | `scripts/promote.sh` invokes `verify_patch.sh` before approval and `verify_target.py` after commit. |
| Promotion requires exact interactive operator approval of the run ID. | `scripts/require_operator_approval.sh` via `scripts/promote.sh`; case T8 covers mismatch rejection. |
| A promotion applies only to `gate/<run-id>`, records a promotion commit, and cleans up failed branch/worktree state. | `scripts/promote.sh`; cases T10–T11 cover a branch and `promotion.json`; failure paths use `cleanup_failed`. |
| Every failure path records a failed `promotion.json` where record writing succeeds. | `scripts/promote.sh` uses `record_status` through `fail_closed`; T5c checks a concrete failure reason. |

### Manual full-health invariants

| Invariant | Current enforcement or evidence |
| --- | --- |
| The full repository health proof starts from a clean source worktree. | `scripts/forge_check.sh` rejects a dirty tree. |
| The full proof runs both existing promotion and runner matrices. | `scripts/forge_check.sh` invokes `tests/promote/run_all.sh` and `tests/runner/run_all.sh`. |
| The full proof is an explicit manual health check, not a configured promotion-time required check. | `README.md`, `docs/v0.2-progress.md`, and `scripts/forge_check.sh`. |

## Test architecture boundary

The architecture has two distinct layers.

1. **Unit-level verifier checks** remain promotion-time checks. They validate a
   candidate patch from its base in a disposable worktree and execute the
   commands configured in `gate.toml`. At present, that is the `unit` check.
   Their job is to decide whether this candidate patch verifies; they do not
   establish that Axiom Forge's runner or promotion harness is healthy.

2. **Contract-level gate checks** are a future, separately invoked test suite
   for the runner and promotion contracts. They may exercise the existing
   runner and promotion scripts from a known-clean developer checkout or
   dedicated CI-style checkout, with deterministic fixtures and explicit
   cleanup. They are not a configured check in `verify_patch.sh` and are never
   invoked by `verify_target.py`.

`forge_check.sh` should remain manual for now. It deliberately requires a
clean source worktree and executes matrices that create runs, temporary
worktrees, and gate branches. That is appropriate for an operator's complete
health proof, but it is not compatible with a promotion candidate that is
applied but uncommitted in the verifier's temporary worktree.

## Non-recursion proof

The future contract suite has one permitted invocation path:

```text
operator or external CI-style caller
  -> future gate-contract test entrypoint
     -> deterministic fixtures
     -> selected runner and promotion matrix cases
     -> cleanup and contract result
```

The prohibited path is:

```text
verify_patch.sh -> verifier command -> forge_check.sh or contract suite -> promote.sh
```

The proof obligation is structural: the future contract-suite entrypoint must
not be listed in `gate.toml`, called by `scripts/verify_patch.sh`, called by
`scripts/verify_target.py`, or reached by any command that either one invokes.
Conversely, the suite may call `promote.sh` only when it is initiated by the
external contract-suite entrypoint, never from a promotion-time verifier.

Therefore a candidate promotion follows only the existing path:
`promote.sh -> verify_patch.sh -> verify_target.py -> configured unit command`.
No edge in that path reaches `forge_check.sh`, the future suite, or another
`promote.sh` invocation. This explicitly excludes designs where
`verify_patch.sh` calls `forge_check.sh` or any script that can call
`promote.sh`.

## Isolation and deterministic fixtures

The future suite should own a dedicated fixture namespace rather than reuse
operator-created runs or branches.

| Fixture | Deterministic design | Cleanup owner |
| --- | --- | --- |
| Source checkout | Start from a clean, explicitly checked contract-test checkout. | External caller before start; suite verifies but does not discard user changes. |
| Agent worktrees | Use unique, suite-owned temporary paths and a per-invocation base SHA resolved at contract-suite start. That base is deterministic within one invocation, not fixed across repository history. | The case that created the worktree, with a suite-level final cleanup guard. |
| Gate worktrees | Let the exercised promotion path create only unique suite-owned temporary paths. | The exercised promotion cleanup on failure/success, plus suite final cleanup verification. |
| Gate branches | Use an unambiguous reserved test run-ID prefix that cannot collide with real runs. | The individual case, with a final suite cleanup guard. |
| Run directories | Generate complete fixture directories under `runs/<reserved-test-id>`. Do not alter real captured runs. | The individual case, with a final suite cleanup guard. |
| Patches and records | Generate fixed textual patches and matching `record.json`/digest fixtures from the current test base. | The case owning its run directory. |
| Logs | Generate minimal deterministic `stdout.log` and `stderr.log` only where a fixture contract requires them. | The case owning its run directory. |
| Verification and promotion artifacts | Treat `verify.json`, `post_verify.json`, and `promotion.json` as case outputs; assert the relevant status/reason before removal. | The case owning its run directory. |

The future suite must not require a clean source worktree inside an
uncommitted candidate-patch worktree. That environment belongs to
`verify_patch.sh` and is intentionally unsuitable for a suite that has a
clean-source precondition. The contract suite's clean-check applies only to
its external source checkout before it begins.

## Reuse versus modeling

The future suite should reuse the existing matrix cases as the executable
contract source, not duplicate their assertions opportunistically.

**Reusable execution cases:** runner R1–R7 and promotion T1–T11 already
exercise the concrete scripts and their recorded reasons, including
parent-dirty handling, adapter state violations, run-directory identity,
stale base rejection, whitespace/apply failure, operator mismatch, existing
branches, and successful promotion records. The existing matrices do not
currently accept a reserved fixture namespace. Before a future suite relies on
matrix reuse, its implementation must first establish namespace isolation for
test run IDs, branches, worktrees, and artifacts; this is a design constraint,
not a current capability.

**Cases to model rather than run through full promotion:** the suite should
model the call-graph boundary itself, the distinction between a clean external
contract checkout and an uncommitted verifier worktree, and the assertion
that no promotion-time command resolves to a contract entrypoint. These are
architecture assertions, not promotion scenarios, and should not invoke
`promote.sh`. It may also model fixture naming and cleanup expectations before
running any mutation-heavy matrix.

Modeling these boundary cases does not alter promotion policy or adapter
trust. It uses synthetic records, patches, logs, and test-only adapters already
present in the repository solely to demonstrate deterministic control-flow and
fixture behavior. It does not promote an adapter, change its status, or treat
test evidence as real CLI-adapter evidence.

## Expected outputs and relationship to manual health

The future contract entrypoint should emit a stable, human-readable result:

```text
GATE_CONTRACT_TEST_MATRIX: PASS
```

On failure it should emit the failed case name, the observed invariant or
cleanup failure, and a non-zero exit code. Its assertions should make relevant
run and promotion artifacts inspectable before fixture cleanup where a failure
needs diagnosis.

`forge_check.sh` remains the existing manual full-health proof and continues
to report `AXIOM_FORGE_CHECK: PASS`. The future contract result is narrower:
it establishes the gate-contract suite's own non-recursive boundary and
selected contract coverage. It does not replace `forge_check.sh`, become a
promotion-time verifier command, or imply automatic promotion.

## Deferred scope

- Promotion policy remains unchanged, including clean-target, current-base,
  exact operator-approval, and gate-branch requirements.
- Adapter trust and adapter status remain unchanged.
- Adapter expansion, including Cursor, OpenCode, CodeBuddy, and similar CLIs,
  is deferred to a later version with separate evidence and approval.
- Signatures, automatic trust or promotion decisions, orchestration,
  dashboards, cloud routing, PR automation, and sandboxing remain out of
  scope.

## Conditional follow-up

If this design is accepted as non-recursive, implement one external-only
`tests/gate_contract/run_all.sh` entrypoint that first asserts the prohibited
call-graph boundary, establishes isolated test namespaces for run IDs,
branches, worktrees, and artifacts, and then invokes the existing runner and
promotion matrices with explicit cleanup checks.
