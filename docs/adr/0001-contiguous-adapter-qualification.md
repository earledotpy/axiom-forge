# Grant standard adapter status from contiguous qualification evidence

Antigravity qualifies for `standard` status and trust after three consecutive independent runs that each satisfy the full adapter-safety contract, produce a verified patch, capture the complete adapter configuration, and pass a deterministic operator-controlled task-specific acceptance test. The independent tasks cover modifying existing behavior, adding a small new behavior, and handling a defined edge case through a dedicated qualification fixture and acceptance harness. A qualification command evaluates one case by capturing the run, verifying the patch, enforcing scope, running external acceptance, capturing configuration identity, and writing a structured result. Each qualification case is a committed directory containing agent-visible task text and operator-controlled allowed paths and acceptance code; only its task text is supplied to the agent. Promotion is not required for this status decision. The acceptance test remains outside the agent worktree and runs only after the patch is applied in a fresh verifier worktree; the patch must also stay within an operator-controlled allowed-path list. A committed qualification report in `docs/adapter-evidence.md` records the run IDs, pinned configuration, task specifications, acceptance and scope results, and patch hashes while raw artifacts remain local under `runs/`. Only after the third successful run may one final approval change update that report, the Antigravity registry entry to `stable` status and `standard` trust, and the required `agy` health preflight. Codex and Claude retain their existing `standard` status under a grandfather clause; new adapters and future changes follow this qualification rule. A change to the adapter-script revision, CLI version, selected model, or relevant configuration invalidates standard trust until a new qualification succeeds. Any failed, unsafe, incompletely identified, task-incorrect, or out-of-scope run resets the series, rather than allowing selective accumulation of successful runs or manual identity assertions, because standard status must reflect a clean reliability signal and leave failures available for separate diagnosis.

Candidate adapter compatibility is a weaker, separate decision. A compatibility
result may show that one candidate CLI can operate through captured-run
creation and patch verification, but it does not satisfy this ADR's contiguous
three-case qualification rule, does not update the adapter registry, and does
not grant or retain `standard` trust. Compatibility evidence may justify
starting qualification; it cannot replace qualification evidence.
