# Adopt a forge package for library modules behind exec'd entry points

Axiom Forge will adopt a real `forge/` Python package as the home for importable shared library modules, keeping `scripts/` for exec'd entry points. The boundary rule is exactly: if it is in `forge/`, import it; if it is in `scripts/`, exec it. `app/` remains the workbench application layer and becomes a consumer of `forge/` rather than a second home for shared domain logic, because `app/` names the interactive workbench and should not also mean "shared library". This replaces the `try/except ModuleNotFoundError` import shim that library modules in `scripts/` currently carry, and it gives the downstream extraction and deepening work a known home for new shared modules.

The shim exists today only because several modules wear both hats: they are imported as libraries and also exec'd by the shell as `python "$SCRIPT_DIR/foo.py"`. Those dual-role modules resolve through a thin exec-shim, the pattern `scripts/workbench.py` already proves against `app/workbench.py`. The real logic moves to `forge/foo.py` as an import-only module with no shim, and a thin `scripts/foo.py` remains as the exec'd entry point that imports `main` from the `forge/` module. Shell call sites do not change. Because a script exec'd by path receives its own directory rather than the repository root on `sys.path`, each `scripts/` entry point that imports from `forge/` must first resolve the repository root and insert it on `sys.path`, exactly as `scripts/workbench.py` already does before importing `app.workbench`. That root bootstrap is what makes the boundary rule work in an exec'd process, and it is the entry point's job rather than the library module's. This preserves ADR-0002's rule that the shell scripts remain the operator-facing interfaces, and follows ADR-0003's direction to move ownership behind shared internals while preserving the operator-facing loop: the shell continues to exec `scripts/*.py` by path and never invokes package modules directly, so `python -m forge.foo` is rejected for this seam. Neither ADR-0002 nor ADR-0003 is amended.

Modules exec'd only by the shell and imported by nothing — the thin structured-access and write helpers such as `json_get.py`, `toml_get.py`, `sha256_file.py`, and the `write_*.py` commands — are entry points and stay in `scripts/`. Modules imported only, such as `concurrent_task_scopes.py`, are library modules and belong in `forge/`. The Bash-to-Python structured-output convention that governs how the shell reads values back from those helpers is a separate decision and is not settled here.

Existing documents name several library modules by their current `scripts/` path — ADR-0002 places run record rules at `scripts/run_record.py`, ADR-0005 and `CONTEXT.md` name `scripts/delegation_artifact_set.py` as the ownership point for delegation-artifact-set rules. Those references identify *which module owns a rule*, not that the module is an entry point, and this ADR does not overturn any of them. Under incremental migration each such path becomes stale only when that module actually moves, so the change that moves a module is also the change that updates the path references naming it, in the same commit. A module path named in an ADR or in `CONTEXT.md` does not pin that module to `scripts/`.

Migration is incremental rather than big-bang. There is no standalone package-migration change; each extraction or deepening change places its module in `forge/` as it lands, and a dual-role module's shim is retired when that module's own change reaches it. The empty package is bootstrapped by the first change that places a module there. Incremental migration is chosen because it matches the repository's behavior-preserving, one-module-per-change discipline and because concurrent work makes a single sweep touching every library module, its tests, and its call sites a needless conflict and review burden. The import shim survives on untouched modules until their change arrives, which is acceptable because it is the existing state and is harmless in place.
