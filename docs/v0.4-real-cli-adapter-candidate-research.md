# v0.4 Planning — Real-CLI Adapter Candidate Research

## Purpose and scope

This is a research-only plan for a dated inventory of possible future real-CLI
adapters. It does not authorize implementation, execution, acceptance, or
trust of any candidate.

The three decisions remain separate:

1. **Candidate inventory** records a product that has enough current,
   first-party evidence to merit investigation.
2. **Adapter implementation** is a separately authorized future change that
   adds an `agents/<name>.sh` script and demonstrates the existing runner
   contract.
3. **Adapter trust and status** are separate human decisions under the
   acceptance criteria in `docs/adapters.md`, supported by captured run,
   verification, and promotion evidence as applicable.

CLI availability, a version command, documentation, or historical evidence
does not prove that a candidate is adapter-safe, compatible, trusted, or
eligible for a status change.

## Preserved v0.3 boundaries

This research must preserve the non-recursive architecture in
`docs/v0.3-non-recursive-gate-contract-design.md` and its implemented
call-graph guard in `tests/gate_contract/run_all.sh`.

- There is no promotion-time gate-contract integration.
- `gate.toml`, verifier scripts, `scripts/forge_check.sh`, promotion policy,
  adapter trust, adapter status, and registered adapters remain unchanged.
- Research does not add adapters, run real CLI experiments, change tests, or
  modify production code.
- Promotion-time verification must not reach `forge_check.sh`, gate-contract
  checks, or `promote.sh` through another path. The promotion path remains
  `promote.sh -> verify_patch.sh -> verify_target.py -> configured unit
  command`.

## Research questions and source hierarchy

For each candidate, establish only what the available evidence supports:

1. Is there a maintained CLI for coding or agentic coding use?
2. Is there current first-party documentation for installation, invocation,
   authentication, permissions, and automation behavior?
3. Does documented behavior leave open a plausible future adapter experiment
   that can satisfy Axiom Forge's supplied-worktree and Git-state contract?
4. Which material questions remain unknown and must be resolved before an
   implementation proposal?

Use sources in this order:

1. Official product documentation, including official CLI references and
   release notes.
2. The owning vendor's or project's official source repository and release
   history.
3. First-party package registries or installation references maintained by the
   vendor/project.
4. Secondary sources only to locate primary evidence; do not treat them as
   compatibility proof.

Record the retrieval date and URL for every source. When primary evidence is
absent, stale, contradictory, or does not answer a question, record the field
as **unknown** or **deferred** rather than inferring compatibility.

### International and non-English primary sources

Non-English first-party sources are valid evidence. For each such source,
record its source language, retrieval date, original-language title, and any
translation notes alongside the URL. Translation may clarify a source for the
inventory, but it must not turn an ambiguous statement into a compatibility
claim the original source does not support.

International and non-Western candidates use the same inclusion/exclusion
criteria and the same evidence record as every other candidate. Model or API
capability alone is insufficient for adapter candidacy: the inventory still
requires first-party evidence of a maintained coding-oriented CLI and must
investigate its invocation, automation, permission, working-directory, and
Git-state behavior.

## Candidate inclusion criteria

Include a candidate in the dated inventory only when the research can identify:

- a real, maintained CLI intended for coding or agentic coding work;
- a current first-party source for installation and normal invocation;
- a named vendor, maintainer, or governing open-source project;
- a plausible task-input mechanism and caller-selected project or working
  directory, or an explicitly documented uncertainty about it;
- enough official evidence to investigate permissions, automation, and Git
  side effects; and
- licensing, platform, account, and model/provider prerequisites that can be
  recorded.

Inclusion means research-worthy only. It is neither an endorsement nor an
adapter acceptance decision.

## Exclusion and deferral criteria

Exclude or defer a candidate when it is:

- IDE-only, browser-only, API-only, or lacks a documented usable CLI;
- discontinued, materially inaccessible, or not intended for coding work;
- documented only by unofficial sources;
- unable to identify a maintained owner/project or current installation path;
- documented as requiring an incompatible unmanaged repository lifecycle; or
- too ambiguous about automation, permissions, or working-directory behavior
  to justify a candidate record.

An otherwise interesting product with incomplete evidence may be listed as
**deferred — insufficient first-party CLI evidence**. That is not a negative
technical finding and must not be upgraded by assumption.

## Required real-CLI candidate evidence record

Every dated candidate record must contain the following fields. Fields not
supported by primary evidence must say `unknown` and name the next evidence
needed.

| Field | Required record content |
| --- | --- |
| Product / vendor / project | Product name and accountable commercial vendor or open-source project. |
| Category | Major LLM provider CLI, major third-party coding CLI, or notable open-source coding CLI. |
| Official sources | Documentation, repository, release-note, and CLI-reference URLs used. |
| Source language and translation | Source language, original-language title, and translation notes for each non-English source; record `not applicable` when all sources are English. |
| Retrieval timestamp | Date (and time/zone when material) each source was retrieved. |
| Installation and access | Install method, supported platforms, license, account/subscription/API-key requirements, and known network requirements. |
| Executable identity | Expected executable name, documented version command, and package/distribution identity. |
| Task input | Documented prompt, file, stdin, command, or interactive task-input form. |
| Automation controls | Documented non-interactive, batch, JSON/structured-output, resume, and approval-bypass controls, including whether each is unknown. |
| Permission controls | Documented filesystem, shell, network, confirmation, sandbox, and approval controls. |
| Working-directory behavior | Documented current-directory, project-root, repository-discovery, and explicit-directory behavior. |
| Filesystem and Git behavior | Documented file scope plus shell, Git, branch, commit, and `HEAD` behavior; distinguish documented defaults from unverified behavior. |
| Outputs | Documented stdout/stderr/log/artifact locations, exit-code behavior, and machine-readable output where available. |
| Model, provider, and routing | Model/provider selection, credentials, remote data-routing implications, and policy-relevant unknowns. |
| Preliminary adapter fit | `research-ready`, `deferred`, or `excluded`, with evidence-bound reasoning only. |
| Unknowns / next evidence | Contradictions, missing facts, and the precise primary documentation or future experiment needed. |

This research record complements, but does not replace, the real-run evidence
format in `docs/adapter-evidence.md`. A future implementation must record
run-bound CLI observations and verified outcomes separately.

## Research procedure

1. Start a bounded inventory organized by the three categories in this brief.
2. Apply the inclusion and exclusion criteria before collecting detailed
   behavior claims.
3. Locate current primary sources and record their retrieval dates; use
   secondary pages only as leads to those sources.
4. Complete the evidence record for each included candidate without invoking
   its executable or creating an adapter script.
5. Compare documented behavior to the existing adapter contract without
   assuming undocumented behavior is safe.
6. Classify each candidate as `research-ready`, `deferred`, or `excluded`.
7. Produce a dated inventory and a small, evidence-backed shortlist for a
   separately authorized future adapter-design task. Do not alter the adapter
   registry or trust/status as part of this procedure.

- Default versus bypassed approval behavior.
- Machine-readable output stability and exit-code semantics.
- Log, session, cache, and artifact paths.
- Authentication-token storage and routing implications.
- Compatibility with Axiom Forge's runner contract.
